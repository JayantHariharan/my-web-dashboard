Overview

Current API Surface

The current backend is intentionally narrow. The frontend auth page talks to these endpoints directly, and the broader app/game platform APIs are not part of the active backend scope yet.

POST /api/auth/login
POST /api/auth/signup

Auth Model

How Authentication Works Today

  • The frontend stores a lightweight client session in sessionStorage and mirrors it in localStorage for restore.
  • /api/auth/me is a username-based lookup, not full token auth.
  • Passwords are combined with SECRET_KEY and hashed with adaptive password schemes.
  • The backend prefers bcrypt_sha256, then bcrypt, and falls back to pbkdf2_sha256 if needed.
  • Local SQLite startup can recover to data/playnexus-recovered.db if the primary file is unhealthy.
  • Keep SECRET_KEY stable per Render environment or existing account verification can break.
  • All auth endpoints share the same in-memory rate limiter.
POST /api/auth/login

Authenticates a user and returns the username plus numeric user ID.

Request Body

FieldTypeNotes
usernamestring1-100 chars, must contain at least one letter
passwordstring1-100 chars
{
  "username": "player123",
  "password": "SecurePass123!"
}

Success Response

{
  "message": "Login successful",
  "username": "player123",
  "user_id": 1
}

Typical Errors

  • 404 No user found
  • 401 Invalid username or password
  • 429 Rate limit exceeded
  • 500 Authentication service unavailable
POST /api/auth/signup

Creates a new account, hashes the password, and records signup IP metadata.

Request Body

FieldTypeNotes
usernamestring3-100 chars, letters required
passwordstringMinimum 8 characters
confirm_passwordstringMust exactly match password
{
  "username": "newplayer123",
  "password": "SecurePass123!",
  "confirm_password": "SecurePass123!"
}

Success Response

{
  "message": "Signup successful",
  "username": "newplayer123",
  "user_id": 2
}

Typical Errors

  • 409 Username already exists
  • 422 Validation error
  • 429 Rate limit exceeded
DELETE /api/auth/account

Permanently removes a user after verifying username and password. The current frontend sends the username and password, and the backend also accepts an optional confirm_username field for compatibility.

Request Body

FieldTypeNotes
usernamestringRequired
passwordstringRequired
confirm_usernamestringOptional compatibility field
{
  "username": "player123",
  "password": "SecurePass123!"
}

Success Response

{
  "message": "Account deleted successfully",
  "username": "player123",
  "user_id": 1
}

Typical Errors

  • 401 Invalid username or password
  • 500 Unable to delete account
GET /api/auth/me?username=<name>

Returns the current user profile in the temporary username-based session model. This is a transitional endpoint until token-backed auth is added.

Query Parameters

ParameterTypeNotes
usernamestringRequired in the current implementation

Success Response

{
  "id": 1,
  "username": "player123",
  "created_at": "2025-03-29T10:30:00",
  "last_login_at": "2025-03-29T15:45:00",
  "profile": {}
}

Typical Errors

  • 401 Not authenticated
  • 404 User not found
GET /health

Checks whether the API is up and whether the database connection succeeds.

Healthy Response

{
  "status": "healthy",
  "service": "PlayNexus API",
  "database": "connected",
  "environment": "production"
}

Notes

  • Used by deployment checks and uptime monitoring.
  • Returns 503 when the database is unavailable.
  • Not rate limited in the current app configuration.
Data Model

Current Tables

The active backend only depends on the auth-related tables below. Table names may gain an environment suffix like _test or _prod.

users

  • id: primary key
  • username: unique login name
  • password: adaptive hash plus pepper
  • created_at: signup timestamp
  • last_login_at: most recent login time
  • created_ip: signup IP
  • last_login_ip: latest login IP

user_profiles

  • user_id: linked user ID
  • display_name: optional display name
  • bio: optional biography
  • preferences: JSON-like settings payload

schema_version

Tracks applied migration scripts.

Security Foundations

  • SQL Injection Hardened: Dynamic SQL builds enforce strict `isalnum()` plus underscore filtering on column schemas and ordering inputs, alongside full dynamic parameterization (`%s`/`?`).
  • Safe DB Migrations: Migration tooling is built to fully mask and sanitize connections, keeping credentials entirely out of CLI output and exception logs.
Quick cURL

Fast Manual Checks

# Signup
curl -X POST http://localhost:8000/api/auth/signup \
  -H "Content-Type: application/json" \
  -d '{"username":"testuser","password":"Test1234","confirm_password":"Test1234"}'

# Login
curl -X POST http://localhost:8000/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"testuser","password":"Test1234"}'

# Current user lookup
curl "http://localhost:8000/api/auth/me?username=testuser"

# Delete account
curl -X DELETE http://localhost:8000/api/auth/account \
  -H "Content-Type: application/json" \
  -d '{"username":"testuser","password":"Test1234"}'

# Health
curl http://localhost:8000/health
